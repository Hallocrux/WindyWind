1. 使用中文。
2. 如果你对这个项目不了解，先阅读根目录 `PROJECT.md`。`PROJECT.md` 只记录当前仍有效的项目状态、默认入口、稳定结论和 coding agent 后续开发确实需要知道的工程约定。
3. `README.md` 面向项目使用者，只写当前已经可用的功能、运行方式、输入输出位置、已确认的限制和已完成探索的入口；不要把每一个软件实现计划、候选方案、分阶段重构细节写进去。
4. 详细设计、实现计划、重构方案、待办拆分、方法教学说明等冗余内容，统一写入 `Docs/` 下按主题拆分的文档，而不是继续堆在 `PROJECT.md` 或 `README.md`。
5. 更新规则：
   - 项目有了新的已实现功能、探索产物或稳定结论时，补充 `PROJECT.md`；
   - 面向用户可见、可运行、可复现的变化，同步更新 `README.md`；
   - 如果新增内容主要是计划、设计讨论或实现草案，写入 `Docs/`。
6. 所有“结论 / 事实 / 判断”都必须显式标注时间，避免旧结论被误读成当前结论。
   - 在 `PROJECT.md` 中，只记录“当前仍有效”的结论，并使用绝对日期，例如 `2026-04-05`；
   - 历史结论、被替代结论、阶段性判断，统一写入 `Docs/`，不要继续堆在 `PROJECT.md`；
   - 对结论文档，优先使用统一字段：
     - `状态：current / historical / superseded`
     - `首次确认：YYYY-MM-DD`
     - `最近复核：YYYY-MM-DD`
     - `数据范围`
     - `代码口径`
     - `证据入口`
     - 如已过时，补 `替代关系`
   - 避免使用“目前 / 现在 / 最近 / 这次”这类相对时间词；如必须表达，改成带绝对日期的句子。
7. `PROJECT.md` 的职责是“当前有效状态卡片”，只保留：
   - 项目定位与默认入口
   - 当前生效的数据组织与工程约定
   - 当前仍有效的稳定结论
   - 指向 `Docs/` 的专题入口
8. 推荐的文档拆分方向：
   - `Docs/data_catalog.md`：数据目录、manifest、工况资产、已知异常字段
   - `Docs/table_pipeline.md`：表格主线的数据处理、特征、train/eval/inference 规则
   - `Docs/data_quality_and_findings.md`：带时间戳的稳定事实与实验结论
   - `Docs/video_pipeline.md`：视频 RPM 与手工标注子项目现状
   - `Docs/experiments/`：每个探索的详细记录
9. 为了更好地组织规则或表达意思，可以直接修改本文件。
10. 永远不使用 `rm` 或 `del`；如果需要删除文件，必须移动到回收站。
11. `src/try/` 专门用于“探索”类工作。探索具有一次性、验证想法的性质，不应和 `src/current/`、`src/baseline/` 的正式流程混放。
12. 每次新的探索，都必须在 `src/try/` 下新建一个独立文件夹，按顺序编号并使用能表达主题的名字，例如 `src/try/001_fft_frequency_plot/`。
13. 每个探索文件夹至少包含：
   - 一个说明文件，写清楚探索目标、输入数据、运行方式和输出位置；
   - 一个可直接运行的脚本或 notebook；
   - 如有图片、表格等产物，优先写到 `outputs/` 下按同名目录组织，避免把输出和源码混在一起。
14. 探索阶段优先复用现有数据加载和清洗逻辑，除非明确需要故意绕开正式口径做对比；如果绕开，必须在探索说明里写明原因。
15. 如果训练、复用或新增了可持久化模型资产，必须在 `PROJECT.md` 中记录当前仍有效的模型资产入口，至少包括模型权重、归一化参数、元数据或配置文件、对应代码口径、训练数据范围和最近复核日期；过时或仅作为历史对照的模型资产写入 `Docs/`，不要堆在 `PROJECT.md`。
